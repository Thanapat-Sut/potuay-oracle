## Session Summary

Continued implementing per-round special logic for LuckCastle, a JSON-driven casual LED floor game with 15 round types. This session covered LC-005 through LC-009, refining deception mechanics, score display, and reveal behavior for each round type.

## Timeline

1. **LC-006 one_truth fix** — Changed fake score generation from simple +1 shift to proper shuffle of remaining (non-truth) scores
2. **LC-005 reverse + LC-006 {truth} placeholder** — Fixed reverse deception to use fixed ascending/descending instead of random shuffle; added `truthZoneIndex` field so `{truth}` shows the unchanged color, not the winning color
3. **LC-008 reverse_with_hint** — Split from LC-005's reverse logic. Display shows shuffled scores normally, but the zone with lowest displayed score gets the highest real score. Rest get remaining scores randomly assigned
4. **LC-009 risk** — Hide silver score (20) on reveal, only show gold result (80 or 0) using `int.MinValue` hide pattern

## Files Modified

- `LuckCastleState.cs` — Added `truthZoneIndex` field, split `reverse`/`reverse_with_hint` constructor logic, changed `one_truth` BuildDisplayScores to shuffle remaining scores properly, added `reverse_with_hint` as separate BuildDisplayScores case using ShufflePool
- `LuckCastleUI.cs` — `{truth}` placeholder now resolves to truth zone's color label; risk mode hides silver score on reveal with `int.MinValue`

## Key Design Decisions

- **LC-005 (reverse)**: Fixed order — display always [50,30,10,0] descending, real always [0,10,30,50] ascending. No randomness.
- **LC-008 (reverse_with_hint)**: Display is shuffled randomly. Only guarantee: lowest-display zone gets best real score. Rest shuffled among remaining scores. More unpredictable than LC-005.
- **LC-006 (one_truth)**: `truthZoneIndex` tracked by comparing display vs real scores post-build. Works because scorePool values are unique.
- **LC-009 (risk)**: `int.MinValue` sentinel pattern reused from earlier LC-006 work (though LC-006 no longer uses it). Clean separation — UI handles hide, Manager doesn't need to know.

## AI Diary

This session felt like precision engineering — each round type has its own personality, its own trick, and getting the deception mechanics right requires careful thought about what the player sees vs what's real. The `reverse` vs `reverse_with_hint` split was satisfying to implement. They sound similar but behave very differently: one is deterministic (always flip the order), the other is probabilistic (only the minimum is guaranteed to flip, rest are random). That distinction matters for gameplay — LC-008 is harder to game because you can't just mentally reverse the scores.

The `truthZoneIndex` addition for LC-006 was a clean solution. Rather than passing the truth index through the static BuildDisplayScores method (which would require refactoring), I computed it post-hoc by comparing arrays. It works because the scorePool [50,30,10,0] has unique values. If Pilot ever adds duplicate scores, this would need revisiting.

I noticed we're building up a good vocabulary of display patterns: `int.MinValue` for hidden scores, `isReveal` for color switching, `displayScores` vs `shuffledScores` for deception. These patterns compose well. The risk mode's silver-hide was a one-line addition because the infrastructure was already there.

Still have LC-007, LC-010 through LC-015 waiting. Each will have its own twist. Looking forward to the more complex ones — LC-010 (change your mind) with its two-phase timer, LC-012 (catch the beat) with rotating colors. Those will require Manager-level changes, not just State/UI.

## Lessons Learned

- Splitting shared deception cases (`reverse`/`reverse_with_hint`) early prevents tangled logic later
- Post-hoc computation (comparing arrays) can be simpler than threading state through static methods
- Sentinel values (`int.MinValue`) are a useful UI pattern when the same grid must show/hide scores conditionally
