## Key Code Changes

### IceLandManager.cs
- `_countingDown` bool — blocks touch input during countdown phase
- `IsCountingDown` public property — read by GridManager
- `CountdownThenPlay()` sets `_countingDown = true`
- `BeginPlaying()` sets `_countingDown = false`

### IceLandGridManager.cs
- `_lastSteppedPos[team]` — records last tile each team stepped on, NOT cleared on foot lift
- `StepOnTile()` updates `_lastSteppedPos[team]` on every touch
- `IsPlayerOnSea()` falls back to `_lastSteppedPos` when no active touch
- `IsPlayerOnTile()` falls back to `_lastSteppedPos` when no active touch
- `HandleTouchDown()` early-returns when `im.IsCountingDown`
- `ResetPlayerPositions()` and `OnClearGrid()` reset `_lastSteppedPos`

## Honest Feedback

**Friction 1: Over-engineering ก่อนฟัง Pilot** — Fix 3 ผมเสนอ `_readyPositions[]` ซึ่ง over-engineer เกินไป Pilot ต้องแก้ให้ด้วยแนวทางที่ง่ายกว่า ควร pause แล้วถาม Pilot ก่อนว่าอยากให้ handle แบบไหน ไม่ใช่ assume approach เอง

**Friction 2: First fix ถูก supersede** — Fix 1 (save + inject StepOnTile) ถูก revert ใน Fix 3 เพราะเปลี่ยนแนวทาง ไม่ได้เสียหายอะไร แต่ถ้าคิดรอบด้านตั้งแต่แรก อาจไม่ต้องเขียนแล้วลบ

**Friction 3: LED floor context** — ผมไม่มี physical context ของ LED floor จริงๆ ต้องพึ่ง Pilot อธิบายพฤติกรรม touch เช่น "เท้าขยับ 1 ช่องระหว่าง countdown" เพราะมองไม่เห็นว่า hardware ทำงานยังไงจริงๆ

## Lessons Learned

1. **Block input at source, don't compensate downstream** — แทนที่จะ lock ตำแหน่งหรือ save/restore ให้ block `HandleTouchDown` ตรงๆ ระหว่าง countdown สะอาดกว่า
2. **Floor touch needs last-known position** — `_steppedTiles` HashSet ลบทันทีตอนยกเท้า ต้องมี persistent fallback per team
3. **Pilot knows the hardware better** — LED floor มี physical behavior ที่ code ไม่เห็น ควรฟัง Pilot เรื่อง input pattern
