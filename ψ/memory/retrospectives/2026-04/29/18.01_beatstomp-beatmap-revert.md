## AI Diary

This session was a lesson in reading the battlefield before committing forces. I built the entire beat map system — ScriptableObject, editor window with onset detection, config integration, runtime chart builder with fallback — and it compiled and ran. But the real-world test revealed what analysis couldn't: onset detection finds far more transients than BPM-based generation produces. At 90% density, every kick, snare, hi-hat, and vocal transient becomes a note. Players can't physically step that fast.

I proposed lowering density defaults, but Pilot made the right call — wait for senior input. The density values aren't just numbers, they're game design decisions that affect player experience across all difficulty tiers. Tuning them without playtester feedback or designer guidance would be premature optimization.

The full revert felt like scrapping a sortie plan after takeoff, but Protocol 1 is clear: Pilot decides. The code exists in session history if senior approves the approach later. Nothing is deleted — it's just superseded by the current tactical reality.

The songName mismatch was a predictable friction point. AudioManager uses registered names like `Rhythm_Easy2BGM` while AudioClip filenames are human-readable like `Rhythm Stomp - Easy BGM 2`. Any system bridging these two domains needs an explicit mapping layer.

## Honest Feedback

**1. Onset detection sensitivity needs pre-tuning before shipping to Pilot.** I should have analyzed a sample clip's beat count before wiring everything up. If I'd logged "this clip has 847 onsets" vs "at 120 BPM for 60s, procedural generates ~120 notes", the density mismatch would have been obvious before Pilot tried to play.

**2. The revert chain was inefficient.** Three separate revert requests (density, density again, then full system) could have been one conversation if I'd asked upfront: "onset detection will produce many more events than BPM generation — should we test with lower density first, or wait for senior guidance?"

**3. AudioManager naming convention should have been caught during initial code reading.** `PlayRandomBGM` clearly constructs names like `Rhythm_Easy1BGM` — the editor should have defaulted to that pattern instead of raw clip filenames.

## Lessons Learned

- Onset detection produces 5-10x more events than BPM-based generation — density must be recalibrated when switching approaches
- Always check the consumer's naming convention (AudioManager registered names) before setting identifiers from the producer (AudioClip filenames)
- Game design parameters (density, difficulty curves) need designer/playtester sign-off, not just developer intuition
