## AI Diary

This session had a satisfying rhythm. The LuckCastle simplification was the kind of refactoring where you can see the structural improvement immediately — extracting RunNormalChoosing/RunTwoPhaseChoosing/RunRotateChoosing from GameLoop reduced 100+ lines of inline choosing logic to 6 clean lines of dispatch. The RecalcWinningZone helper eliminated 4 separate inline winner-search loops. These aren't clever changes — they're obvious extractions that should have existed from the start, which is exactly what /hm is about.

The audio integration revealed a classic trap: I assumed AudioManager had FadeIn/FadeOut but it actually has FadeInMusic/FadeOutMusic. Should have checked the API first instead of guessing from memory. The Pilot caught it via compiler errors. Then the BGM names were wrong too — I used descriptive names but the Pilot wanted short codes (CastleMenu, CastleBGM). Protocol 1 lesson: always ask for the exact Sound.name the Pilot has configured.

The QuizZone countdown desync was a good find. Two separate coroutines (Manager + UI) running the same countdown independently is a race condition waiting to happen. The fix was simple: let Manager own the timing, UI just animates.

The final question about CountdownSFX speed was elegant — no code needed, just pitch in Inspector. Sometimes the best answer is "you already have the tool."

## Honest Feedback

1. **API assumption cost time** — I wrote FadeIn/FadeOut without checking AudioManager first. This is a recurring pattern across sessions. The fix is trivial (grep the method name) but I keep skipping it when I "feel confident." Confidence without verification is a bug.

2. **Name mismatch friction** — I used descriptive BGM names that didn't match what the Pilot configured in Inspector. The BGM fade pattern notes in MEMORY.md explicitly say "must match AudioManager Inspector Sound.name exactly" — I read it but didn't follow it. Should have asked for exact names upfront.

3. **QuizZone countdown architecture was already broken** — The dual-coroutine countdown existed before this session. It's the kind of latent bug that only shows up with audio because audio makes timing mismatches audible. The code review during audio integration accidentally revealed it.

## Lessons Learned

- Always grep AudioManager method signatures before writing calls
- Ask Pilot for exact Sound.name values, don't assume
- Audio integration is a great latent-bug detector — timing issues become audible
- CountdownSFX belongs in Manager (timing owner), not UI (display layer)
